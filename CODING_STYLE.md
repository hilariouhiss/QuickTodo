# Quick Todo 编码风格与开发规范（v1.0）

## 1. 总体原则
- 优先级：Qt 官方风格 > 本项目约定 > 个人习惯。
- 默认遵循：SOLID、KISS、DRY、RAII。
- 新代码必须满足可读、可测、可维护，禁止“能跑即可”式实现。
- 行为改动与风格改动应分开提交，避免评审噪音。

## 2. C++（Qt）规范
- 统一使用 C++23，不引入与项目标准冲突的旧写法。
- 命名规范：
  - 类名：`PascalCase`
  - 函数/变量：`camelCase`
  - 成员变量：`m_memberName`
  - 常量变量：`kPascalCase`（小写 k 开头，后续单词首字母大写）
  - 宏定义：`MACRO_CASE`（全大写，下划线分隔）
- 常量使用细则：
  - 优先使用 constexpr 替代宏定义。
  - Qt 字符串常量优先使用 QStringLiteral 以减少运行时开销。
  - 类内静态常量应在头文件中声明（使用 static constexpr），避免在源文件中重复定义。
- 资源管理遵循 RAII；禁止手写裸 `new/delete`（Qt 父子对象模型除外）。
- 头文件最小包含，优先前向声明以降低编译耦合。
- `#include` 顺序统一为：
  1. 本文件头
  2. 本模块头
  3. 第三方/Qt 头
  4. STL 头

## 3. Qt 约定
- 需要元对象能力时必须声明 `Q_OBJECT`。
- 对外状态优先通过 `Q_PROPERTY` 暴露，并配套 `NOTIFY` 信号。
- 信号命名使用状态变化语义（如 `tasksChanged`）。
- 跨层通信优先信号槽，不直接跨层依赖实现细节。
- QML 调 C++ 的边界保持返回值可预测，不允许异常直接穿透到 QML。

## 4. 错误处理规范
- 统一模式：返回值/状态 + 错误消息 + 日志。
- 底层错误保留技术细节，上层输出用户可读语义。
- 不吞错：失败路径必须可观测（如 `lastDbError`、`errorOccurred`）。
- 异常仅用于不可恢复场景；边界层必须 `catch` 并转为状态。

## 5. QML 规范
- `Main.qml` 只做主窗口壳层（窗口、导航容器、全局弹窗）。
- 业务页面放 `pages/`，可复用组件放 `components/`。
- 页面层仅做绑定与事件转发，不承载复杂业务逻辑。
- 命名规范：
  - 组件文件：`PascalCase.qml`
  - `id`：`camelCase`
  - 属性：语义化命名，避免缩写歧义
- 同一页面布局策略保持一致（优先 `Layout`，必要时局部 `anchors`）。
- 避免魔法数字，统一提取为命名属性常量。

## 6. 架构分层约束
- `view`：展示与交互。
- `viewmodel`：状态编排与命令协调。
- `model/repository`：领域模型与数据访问。
- `infrastructure`：日志、数据库等基础能力。
- 分层依赖方向必须单向，禁止反向依赖。

## 7. 格式与工具
- C++：建议使用 `clang-format`（Qt/LLVM 风格为基线，按项目规则微调）。
- QML：建议使用 `qmllint` + `qmlformat`。
- Qt 静态检查：建议启用 `clazy`（至少 warning 级别）。
- CMake：统一缩进与块风格（如 `if (...)` / `endif ()` 保持一致）。
- CI 建议至少包含：构建、QML 检查、静态分析。

## 8. 提交与评审门禁
- 提交前必须通过：编译 + 诊断检查 + 最小运行验证。
- 变更说明必须包含：改动范围、行为变化、风险点、验证步骤。
- 禁止提交：临时代码、长段注释掉的旧逻辑、未使用文件。

## 9. 适用范围与执行
- 本规范适用于当前仓库所有 C++/QML/CMake 改动。
- 新增模块默认遵循本规范；偏离需在评审中明确说明理由。
